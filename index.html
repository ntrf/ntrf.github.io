HxCPP wrappers without C++

I like using Haxe. It’s a modern language with type inference, GC and a very powerful macro system. It’s also portable to everything and has a decent toolset (although debugging leaves a lot to be desired).

But C/C++ is what every library uses

If you take a normal vanilla Haxe, it can’t link with anything, because it’s execution environment is pretty much undefined. I’ll focus on HxCPP instead, because it gives us a concrete environment to work with.

When Haxe code is compiled for HxCPP target, it will generate C++ sources and a project in a form of .xml file. Then generated sources get compiled by C++ compiler and linked into an executable. So naturally, as Haxe gets converted into C++ you should be able to interface with external C++ code, right?

The OpenFL way

As you can see in many cases of OpenFL Extensions, they typically have:

  • a pre-compiled part as .ndll files,
  • which are generated from the HxCPP project, stored in /project.

On one side, presence of .ndll files allows using them in both HxCPP and Neko … although i can’t imagine why would you ever want to use Neko in the first place. On the other side, this requires compiling “extension” for each platform, but it has to be done as an offline process during first installation (or first use). This makes it difficult to update a native version of the library as you’ll have to recompile many different variations. Library interface is also limited to whatever objects Neko VM can accept, and C++ side of the library has to do all the marshalling, which requires a lot of work if native library ever gets updates. For example Neko VM has two ways of expressing integers: short integers are limited to 31 bit signed numbers, but long integers can use 32 bit space, but are allocated values. Finally, the .ndll file is external to the application, making it impossible for C++ compiler to remove the functionality, that is not referenced.

Anyway, while this approach is straight-forward, it has many drawbacks.

Linc Way

Linc is not a library, but a framework for making libraries. What it does is it adds custom .xml file into the generate project .xml, allowing any C++ code to be added into project source. This eliminates the need for a separate project and just adds all of the external C++ sources to the main project. As a result:

  • C++ compiler and Linker can remove unreferenced functions (you need to use -ffunction-sections flag for that),
  • Linker can embed all of the code into your application, making it smaller,
  • No code for dynamic library loading is needed,
  • Any C++ types, which Haxe can express, can be used in the wrapper as well, including Int64.

Obviously, it looses compatibility with Neko, … but … that’s not even a downside, considering that Neko is surpassed by both eval and HashLink.

For me it was clear, that Linc is a better way of making native extensions. But maybe we can do even better?

Don’t use C++

Let’s take one Linc library as an example: linc_nanovg.

Yes, it’s a windows console. And no, we won’t discuss it today.

nanovg directory contains Haxe side for the wrapper. lib directory is where NanoVG source code itself lives. Some Linc projects have Git submodules linked, instead of linking the source. linc directory needs a closer look:

Linc.hx is where .xml injection macro lives. For some reason it’s not shared among different projects. linc_nanovg.xml is what gets injected into the C++ project. We also see a number of C++ files, which are native side of a wrapper. This last bit is what i would like to avoid.

Creating my own wrapper

Note: I’m using Haxe 4.1 . Older versions might not have all of the described functions.

I wanted to use one library – GLFW – for which there was no wrapper available. I didn’t want to create C++ files for my wrapper, only Haxe files. Is it possible? As it turned out – almost.

Data Types

In order to make in happen we need to know how to pass data to C++ and back. Let’s start with data types:

cpp.Void exists as a substitute of C++ void keywords, but it’s not going to be necessary, unless you start declaring function pointers. The reason why standard Void type from haxe std library can’t be used is because Haxe type is allowed to have value of null, where in C++ void can’t have any value. This does not affect statically typed invocations, but it does make a difference for reflection. When compiling, Haxe backend for HxCPP will determine if the function can be declared with a true void type or it will have hx::Void return type instead. In most cases i’ve used Void as return type and it works as expected.

Our next stop is enums. We have two ways of dealing with them:

  1. Define a normal enum abstract with values taken from C++. This is an easy way, but it’s only applicable if values aren’t going to change. Until recently sequentially numbered enum values were difficult to use in Haxe, because you had to assign each value manually, but with Haxe 4.0 you can relax and copy the parameter names. All-in-all converting a list of C++ enums for use in haxe takes only some minor editing and a few regular expressions. Here is an example of how i’ve handled it for GLFW and, believe me, i didn’t type all of these constants by hand.

  2. Define native static variables in a class. You want to do this for any enum values or constants, that might have different values depending on target platform (like WSAEAGAIN vs EAGAIN on Linux), or otherwise have unpredictable value. Some libraries, such as zLib have version constants, that have to passed into initialization function in order to verify, that dynamic library matches the header file. In this case we have to use @:native meta to declare native field and let C++ side insert a value instead. For GLFW wrapper i’ve used this method for many constants, even if they didn’t need that.

There are two types of pointers we can use:

  • Raw pointers (cpp.RawPointer and cpp.ConstRawPointer), that will be represented in generated C++ code as real pointers. They can be used as parameters, returned values and even class fields, but they can’t be used in containers, such as Array. They also don’t have any reflection support.
  • Regular pointers (cpp.Pointer and cpp.ConstPointer), that can be stored in arrays, but don’t look like pointers to C++ side, and have to be converted to Raw pointers before being passed as arguments.

Here is an example of using pointers in function parameters. In order to represent strings, Haxe has cpp.ConstCharStar type, which (as the name implies) gets converted to const char * arguments. As usual, here is an example of parameter and return value. Haxe will handle all the casts automatically.

In some cases pointer types are completely opaque, so that classes or structs are declared in header file, but not defined. In this case Haxe does not need to know we’re working with pointers at all. Any Haxe class can then be annotated as native pointer type. As long as class is extern and does not have a constructor, there will be no issues. In these cases type can be used as argument or return type directly.

Declaring structures is a bit more complicated. I haven’t figured out all of the details myself, but there seems to be a way to declare structures with reflection support using cpp.Struct. Luckily, GLFW does not declare any structs, but i had enough fun with that when trying to make a wrapper for Vulkan (which is a topic for another day). The basic layout looks like this:

You can’t create these structures with new operator, but you have assign a value to a local variable before using it, so i’m declaring a special static value to initialize it.

This way i can write var qfp = VkQueueFamilyProperties.make; and it will make both C++ and Haxe compilers happy. If you want to clear C++ structure with zeros, you can specify @:native("{ 0 }") instead.

Keep in mind that these structures can’t be used as a value of Array, because they lack reflection info. To handle that, i’ve made RawArray class, that allows me to allocate memory for structures. Obviously, this requires manual memory management, as garbage collector won’t be able to make any sense of raw memory without reflection. I’m sure there is a better way, but i’m sticking with what i’ve got.

Function calls

Now that we have a means to pass data, we can start declaring functions. For that you need extern class and a bunch of static functions.

@:native meta specifies a real function name, but you can name function on the Haxe side with whatever makes sense. I don’t think it’s great to use C-style functions, when you have Object-oriented language. Callbacks

Ok, this is where that “almost” comes from. Basically, you can declare function pointers in Haxe, that have all the parameters, that C++ expects, and everything works as expected. You can use static functions as function parameters easily. In order to make it possible to use lambdas or methods as callbacks, i had to wrap an entire call in a macro, that stores a Haxe object with handler pointers as “user pointer” inside of GLFW Window. It’s not great, as it introduces an anchor, that will prevent handlers from being garbage-collected while window exists. However, that is the nature of this excersize, as there is no way to use C++ libraries in GCed languages without having to think about memory retention.

What you can’t do

In case of GLFW there will be no issues, because the entire library will be compiled as part of my own project. However, there is a problem with declaring callbacks functions, that will be passes to runtime libraries. The name of this problem – Call ABIs. There is no way you can declare a function in Haxe, that has __stdcall specifier in its C++ version and, as a result, you can’t use callback functions declared with this specifier. I’ve ran into this issue when making a wrapper for Vulkan debug layers. Sadly, there is no way around it. I’m not sure if this issue is Windows-only, but i refuse to use a non-portable solution.

I’m planning on making a patch for Haxe, that will add ABI support to HxCPP backend. It’s not that difficult, especially since ABIs are already declared in standard library and even @:abi meta exists. However, all of that is only used for external functions and function pointers, not for the functions, that Haxe produces.

Linc-GLFW

Now let’s get back to practical side. Repository for linc_glfw is here, for those who following me at home.

First of all, i’ve linked GLFW git repository into my project as a submodule. That way i’m not allowed to make any substantial changes to the original library, so it will be easier to update. It also saves repository space at the expense of some headache for end-users, esp. if they forget to include --recursive flag. In the absence of C++ package manager, i find git submodules to be a perfect choice for linking dependencies.

In order to make GLFW build process work, i had to dig through CMake files, in order to get a list of files for each platform. I didn’t do Linux, because i had no machine to test on anyway. The process is straight-forward if you have any experience working with CMake. I suspect, that it can be automated by effectively adding a fake “HxCPP” cross-compiler target in CMake, but it’ll have to be done with predefined “platforms”, as CMake might perform a lot of configuring for external platform.

The list of all source files from GLTF library gets written in XML, as i’ve already mentioned. For external library files, i’m creating a separate group called linc_glfw. That way none of the flags, applied to the main group are going to affect GLFW sources. Haxe group, which contains all of the Haxe-generated C++ files, only gets defines, that are needed for glfw.h include file. I’m specifying GLFW_INCLUDE_NONE, because i’m going to use linc_opengl and it uses it’s own copy of OpenGL includes coupled with GLEW loader.

There is one more group: __main__, which is unfortunate consequence of how Haxe generates code. My Main class contains fields, that are references to GLFW Window, therefore include file for that class also includes glfw.h. The problem is that startup files, that Haxe also generates (__main__.cpp, __boot__.cpp and __lib__.cpp) also include my Main class, but compiled as part of different group. That compilation will fail, as __main__ group does not have include search paths and defines specified. Adding duplicate flags for that group solves the issue. An alternative solution would be to have simplified Main class, that does not have any references to wrapped types:

This way Haxe would not have any references in generated header file, but i wanted a more “universal” solution.

Well, that’s it. That’s the library. It’s not quite finished yet, but it serves as a nice proof, that you don’t need to write .cpp code in wrappers. I’ll probably make another post describing the library itself, when i get to properly test it’s implementation.

2020-06-24

Is TLS small yet?

Ok, we all know that TLS is fast. At least that’s what one Google-related website is trying to convince us with. While they focus on big servers and big browsers to live happily together, what about smaller ones? What about mobile games, that want to interact with social network API, or x-duinos and x-pies? Is it possible to shrink TLS down to something useable with all of this big infrastructure? Besides, i believe, that software should fit it’s useful data into 32KB and TLS is just a transport.

tinyTLS

Back in 2014 i needed a TLS/HTTPS library. Realizing that:

  • OpenSSL is HUGE and full of bugs,
  • Same with NSS but with less bugs,
  • System-provided API likes to enforce their own will,
  • other SSL libraries try to copy OpenSSL, while being commercial,

i decided that there is no hope and i just have to do it myself.

Note: Most commercial TLS libraries offer GPL licenses, but that’s not practical for mobile software. The fact that commercial software is using OSS license (GPL really) just to slap “We’re Open-Source” on their landing page is a topic for another day.

And the result is tinyTLS. As we were using it in a game to make occasional REST queries to server, performance didn’t matter that much, as network would be slower anyway. But we wanted to use off-the-shelf server software, so no PSK modes and other weird stuff. I did in fact implement client certificates, but never used them, because it wasn’t possible to store them securely within the game.

tinyTLS implements TLS 1.0 RFC2246. It’s only a client side library, so it can have only a minimum set of required stuff. Here is a list of things i had to include:

  • AES128
  • MD5, SHA1, SHA2-256, SHA2-384, SHA2-512
  • HMAC-MD5, HMAC-SHA1
  • Big integer modular exponentiation
  • RSA Encryption, RSA Signature verification
  • RSA Signing, but that’s optional for client certificate support

Notice, i have not implemented RSA Decryption – it’s only required on server. Also i’ve skipped SHA2-224, because nobody uses that. Other versions of SHA2 are used for certificate signature verification.

The only cipher suite implemented was TLS_RSA_WITH_AES128_CBC_SHA. Fast forward 5 years and now we can drop MD5 and HMAC-MD5, which is no longer used in handshake of TLS 1.2.

There are limits

While i was able to trim down the size of executable, tinyTLS still required a lot of dynamic RAM allocations (heap allocations). In fact most of the libraries do. I remember some other library (i can’t remember which one) claiming that they only use 6KB of RAM and therefore suitable for MCU applications. As a “proof” they’ve published linker .map file, in which you could have clearly seen that it does in fact use 6KB of RAM and it also links file with suspicious name ucalloc.c.

In practice there is a limit on how much memory you need. In order to receive, decrypt and verify packets you need at least 16K + 1024 bytes. That’s what standard dictates as a maximum size of a fragment. All fragment processing has to be atomic, so you can’t decrypt part of the fragment and return it to application, because you can’t authenticate it properly.

The other problem is fragmentation. TLS effectively has 4 independent byte-oriented streams of data: CCS, alert, handshake and application. We don’t have any structured data on the last one, so we can return it in whatever fragments we like. The first one is also easy – any time you receive a non-empty fragment, you change ciphers. With alert it’s kinda difficult, as you have to interpret incoming bytes in pairs. In fact there was a recent discovery of an OpenSSL bug, which allows an attacker to inject byte string 21:03:03:00:01:01 into TCP steam right before ServerHello and OpenSSL will remember 01 as an alert byte and keep it around until real alert arrives, in which case it will convert any fatal alert into warning and keep connection open. In my implementation i didn’t bother with that and i always treat alert fragments of less that 2 bytes as a protocol error, and it does seem safer that way.

The worst part is handshake, which has it’s own independent framing inside of TLS fragments. Handshake messages are not aligned with fragments at all. Say you need to receive a certificate and for some valid reasons server is sending it to you in fragments. You need to check previous fragment you’ve received for handshake header (which may be partial btw), get the size of the entire handshake message, allocate memory for it and then receive each fragment, constantly copying data from crypto buffer to handshake message buffer. Now imagine doing the same without memory allocations. Even if handshake messages had their framing aligned with fragments, it would’ve made things much easier. And don’t forget that the original framing of TLS is sent in TCP framing, so we get framing within framing within framing. *sigh*\

Any good news?

At some point someone had a good idea to allow implementations to reduce the fragment size, because TLS was out-of-reach for embedded systems. A typical MCU has about 16KB to begin with and that needs to hold some useful data first, leaving only a little bit available to TLS. That became RFC6066 sect. 4 and later updated with RFC8449. This extension alone allows us to implement TLS with less memory. For my implementation i’ve chosen 512 bytes – not too small, because of the overhead of MAC and not too large to explode the buffer. But we still need to store handshake messages and there is no way to constraint them.

When we’re sending messages, we can fragment them as much as we want, so we can safely assume 512 bytes + 64 bytes of MAC, padding, IV and gap between messages will be available to us, which is much less than what standard requires. 512 bytes is typically enough for a small-scale REST query and you can use blocking socket api to fragment larger messages.

We can store handshake messages in a combined buffer, because both client and server should be taking turns anyway. Most of the handshake messages will be less then 512 bytes, leaving us in the worst case scenario with 2 fragments present simultaneously. The one exception is a Certificate message. More on that later.

So in theory, if you manage to compress all of the state well enough, you can get away with just 1KB + some state for encryption algorithms and padding. Well … that’s theory.

Enter zeroTLS

I took spare parts from tinyTLS, most notably most of the crypto and made zeroTLS. This time an entire context is placed into one continuous block of memory. If you can guarantee that it’s always aligned to 16 bytes, it cen even be moveable. That makes integration with GC languages exceptionally easy – just allocate an array of Simd.Vec4 or whatever and you’re done.

Bu that was not the point. The point was to figure out how much memory some of the popular websites need. If we can fit within 6KB, it’s good enough, but if we can go lower than 4KB, that would be prefect.

I’ve stated with a local nginx install with self-signed certificate. That way i won’t wake up some site admins in the night with some strange OpenSSL warnings in the log. Fortunately modern versions on nginx have support for both RFC6066 and RFC8449.

After i’ve been confident about the results, i let it loose and tried to fetch /robots.txt from some internet hosts.

Results

Published on a separate page.

hostname 24KB 12KB 6KB 4KB 3KB
localhost Nginx ok ok ok ok ok
example.com ok ok ok ok mem
api.github.com ok ok ok ok mem
google.com ok ok ok ok mem
api.digitalocean.com ok ok ok ok mem

mem = ZTLS_ERR_OVERFLOW, out of memory error

4KB worked just fine. 3KB did not … pretty much anywhere. Application protocol was not an issue.

The main issue was to fit Certificate message during the handshake. I’ve managed to receive certificates incrementally, so i didn’t have to fit all the certificates, but only one of them. Keeping an entire certificate in memory later is not an option, but i need to store them. One of the problems that surfaced during development of tinyTLS was the fact that many websites have their certificates in the wrong order. This is clearly against the specification, but there is a lot of stuff against the specification in the wild. As a solution i’ve extracted a public key, a signature, key identifiers, some constraints and a hash of the entire certificate as a separate structure. This way instead of storing 1.5KB certificate i only get to store ~760 bytes. That wasn’t enough. At some point memory will contain some context state variables, running hash of handshake messages (for Finished), session id, random values from Client/ServerHello, received certificate and data from previous certificates. An attempt to allocate memory for a new extracted certificate data would fail and the entire handshake had to be aborted.

Local Nginx did not exhibit this problem with 3KB of memory, because it only has one self-signed certificate. Github was very close, with only about 180 bytes short.

So is it 4KB then?

There is not much you can do to reduce memory usage, unfortunately. Parsing certificates on-the-fly is not an option as you need to compute a hash of the signed part, but you don’t get to know the algorithm before the end of signed part. Computing several hash algorithms at once does not sound like a good idea and will require even more memory. The only part then, that can be left behind is the signature itself, which only takes about 20% of the certificate. Once can hope to verify certificates on-the-fly one after the other, assuming that server does in fact comply with an RFC. However, that not as easy as it sounds, because you need to keep public key from the top certificate (you need it later in the handshake), signature of the previous certificate and currently received certificate, which is still about 2KB, but now you also need some space to store big integers. The advantage of doing it after-the-fact is that you can use space previously occupied by the received certificate.

What would definitely help is smaller certificates. ECDSA certificates would make a huge difference, but until we see wider adoption of ECDSA by intermediate CAs, that not going to make much difference. Certificate compression (draft RFC) wouldn’t help as you’ll have to decompress them somewhere.

BTW: Google’s Certificate Transparency timestamp, which is now included into every certificate doesn’t help with the size either. It’s especially concerning, since it does not add anything to security if client does not have public keys for a particular certificate list. Currently there are 17 of these lists and bundling all of those keys is not an option.

What i think i’ll do in zeroTLS is introduce an additional handshake buffer, which is only going to be available for handshake data and can be returned to the application afterwards. This can make renegotiation implementation difficult, but i’m not if it’s even needed at this point.

There is a slight chance that TLS 1.3 needs less memory, but i doubt that. If anything it looks to be more complex, not less.

Until then, I can get to 4KB, but no further.

P.S.: Google does not support changing fragment size and they claim it’s necessary for performance. Their idea of TLS performance is to allow server to choose TLS fragment size freely depending on network capabilities. Fortunately they use 1400 bytes fragment size by default, so in theory it can still work with 3KB. If only certificates were small enough.

Your computer has 32KB of RAM

*”No, it’s not!”*, you might argue.

Fine, those DDR4 DIMM slots of yours are probably full of high-speed low-drag memory chips. Or maybe you have latest and greatest and applest laptop and all of those chips are soldered onto the motherboard. And look at the amount: 16GB, 32GB, 64GB, oh my. That’s so much memory. Clearly bigger, than 32KB.

So let me clarify:

Over the last 40 years the average size of memory, accessible from CPU within 4 cycles, has always been 32KB.

Yes, your computer might have memory, capable of storing 16GB worth of moving pictures, while you edit them, but only a fraction of that data is available to CPU without much delay. That’s what L1 cache is for (sometimes there is also L0 cache, but i think L0 is a store buffer, which is bit different). Anytime you access data beyond size of your L1 cache, you’re waiting for it. In most cases compilers, out-of-order execution units inside CPU and thread schedulers can’t optimize away delay of L2 request, which could be on a scale of 10-50 cycles. Main memory is even slower. Also L2 is typically physically indexed, which means that there will be an additional TLB translation delay or even pagetable load if virtual memory is in use. Some modern CPUs offer memory encryption, which further increases main memory.

*”But main memory is not slow, it’s ~16GB/s.”*

True, but that’s throughput, not latency. It’s still not accessible within 4 cycles. A typical DDR4 PC4-12800 memory has clock speed of 800MHz with 14ns CAS latency (11-11-11 “timings”). That means it’ll take ~25 cycles (on avg.) to access our data, which gives us ~31.25ns access time or 1 cycle at ~32MHz. You CPU my be 3.2GHz, so it’s 100x faster than memory. It’s probably going to take more time for memory to reach individual CPU in multicore system as multiple buffers, busses and crossbars will propagate it from DDR4 interface, so we can effectively multiply it by 2.

If we go back in time and look at what was available 35-40 years ago, you’ll see that things didn’t change that much. I’m going to use ZX Spectrum™ 48 as an example. It had 16KB to 48KB of memory and all mof it was available in one cycle. Well almost, Z80™ CPU works in 4 clock units, during the first two cycles it fetches instructions and then it executes them. So effectively 1 CPU cycle = 4 clock cycles. Memory was running as 2x speed, serving both the CPU and a simple DMA for video generation. Clock speed was ~1.75MHz for memory and twice as low for CPU (oh those weird old days of CPUs being slower than memory). Depending on how you cont cycles it’ll be either 1 cycle access or 4 cycle access latency.

Lets use NES™ as another example. Depending on the region, MOS 6502 processor would have single-cycle access to memory at either 1.79MHz or 1.66MHz. The total size of memory was only 2KB but still – single cycle. Commodore™ 64? Sure, 64KB at 1 cycle access (it also has 6502). Apple™ II? 4KB-64KB at single cycle.

As memory sizes started to grow, CPUs had to use caches, 4KB, 8KB, because it takes a lot of effort to make memory fast as its size increases. Internal latencies from decoder logic, access logic, dynamic logic pre-charge. Because physics. Therefore the memory closes to CPU was always within 8KB-64KB. Some systems took it to extreme: Sony™ PlayStation™ only had 1KB or fast RAM. What was originally a cache ended up being locked to a certain region. And people made GAMES for that platform. In fact, people have been making games for ALL OF THEM.

That should give you a perspective of how bad modern programming practices are. Your computer or computer that you’re targeting with your software, regardless of its age, cost or manufacturer identity, always had, have now and will have 32KB of memory. Anything beyond that should be treated as a long-term storage. If your software needs more than 32KB to be present in memory at all times – it is a bad and inefficient software.

The End.
